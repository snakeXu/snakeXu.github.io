class Game extends React.Component {
  constructor(props){
    super(props);
    this.state = {
      history:[{
        squares:Array(9).fill(null)
      }],
      stepNumber:0,
      xIsNext:true,
    }
  }
  jumpTo(step) {
    this.setState({
      stepNumber: step,
      xIsNext: (step % 2) === 0,
    });
  }
  handleClick(i){
    const history = this.state.history.slice(0, this.state.stepNumber + 1);
    const current = history[history.length - 1];
    const squares = current.squares.slice();
    if (calculateWinner(squares) || squares[i]) {
      return;
    }
    squares[i] = this.state.xIsNext ? 'X' : 'O';
    // console.log(squares)
    this.setState({
      history: history.concat([{
        squares: squares,
      }]),
      stepNumber: history.length,
      xIsNext:!this.state.xIsNext,
    })
  }
  render() {
    const history = this.state.history;
    const current = history[this.state.stepNumber];
    const winner = calculateWinner(current.squares);

    const moves = history.map((step,move)=>{
      const desc = move ? 'Go to move #' + move : 'Go to game start';
      return (
        < li key = {move}>
        < button onClick={() => this.jumpTo(move)}>{desc}< /button>
        < /li>
      )
    })

    let status;
    if (winner) {
      status = 'Winner: ' + winner;
    } else {
      status = 'Next player: ' + (this.state.xIsNext ? 'X' : 'O');
    }
    return (
      < div className="game">
        < div className="game-board">
          < Board 
          squares = {current.squares}
          onClick = {(i)=>{this.handleClick(i)}}
          />
        < /div>
        < div className="game-info">
          < div>{status}< /div>
          < ol>{moves}< /ol>
        < /div>
      < /div>
    );
  }
}

function calculateWinner(squares){
  const lines = [
    [0, 1, 2],
    [3, 4, 5],
    [6, 7, 8],
    [0, 3, 6],
    [1, 4, 7],
    [2, 5, 8],
    [0, 4, 8],
    [2, 4, 6],
  ];
  for(let i=0;i < lines.length;i++){
    const [a,b,c] = lines[i];
    if (squares[a] && squares[a] === squares[b] && squares[a] === squares[c]) {
      return squares[a];
    }
  }
  return null;
}

网易微专业福利期-2.1-JavaScript内存管理

1.前端为什么要关注内存
任何一个程序的运行都要分配内存空间，对于一个页面来说，如果一些不需要使用的内存没有得到及时释放，我们称这种现象为内存泄漏。一次内存泄漏似乎不会有太大影响，但是内存泄漏的堆积会造成内存的溢出(内存溢出简单来说就是我们所需要使用的内存空间大于可用内存，此时我们的程序就会出现内存溢出错误)，在浏览器的表现就是浏览器长时间无响应或者浏览器崩溃。作为前端程序员我们需要给用户良好的用户体验，因为我们需要
·防止页面占用内存过大，引起客户端卡顿甚至无响应
·Node.js使用V8引擎，内存对于后台服务的性能至关重要，因为后端服务的持久性，后端更容易造成内存溢出
2.js数据类型与js内存机制
2-1.数据类型
·原始数据类型:字符串(string)、数字(number)、布尔(boolean)、空对象(null)、为定义(undefined)、symbol
·引用数据类型:object
·内存空间:栈内存(stack)、堆内存(heap)
JavaScript原始数据类型都有固定的大小，保存在栈内存中，由系统自动分配存储空间我们可以进行操作，因此原始数据都是按值访问的。栈是一种运算受限的线性表，其限制是仅允许在表的一端进行插入和删除运算。
var a=10;这个赋值过程都发生了什么？首先，将10存入内存空间，之后在我们当前的作用域中声明一个变量a,此时a的值为undefined，然后将变量a与我们刚才存入内存空间的10关联，变量定义到此结束。
js中的引用数据类型大小是不固定的，存储在堆内存空间，js不允许直接访问堆内存，我们操作对象时是在操作对象的引用而不是操作实际对象。我们可以粗浅理解操作引用就是操作保存在栈内存中的一个内存地址，该地址与堆内存的实际值相关联。引用数据的值是保存在堆内存中的对象。
当我们定义一个函数的时候，会在堆内存先开辟一个内存空间，将函数体中的代码以字符串的形式存储到内存中去，这块空间会有一个十六进制的内存地址，会把这个内存地址赋值给函数名(此时函数名与内存地址类似于原始数据类型存储在栈内存中)
3.垃圾回收
JavaScript的垃圾回收机制
垃圾回收:找出那些不再继续使用的变量，然后释放其所占用的内存，垃圾回收器会按照固定的时间间隔周期性的执行这一操作。(也就是说垃圾回收是自动执行的、不可见的)
JavaScript使用垃圾回收机制来自动管理内存，垃圾回收是一把双刃剑:优势——可以大幅简化程序的内存管理代码，降低程序员的负担，减少因长时间内存运转带来的内存泄漏问题；不足——意味着程序员将无法掌控内存，JavaScript没有暴露任何关于内存的API，我们无法强迫其进行垃圾回收，更无法干预内存管理。
3-1.垃圾回收策略
3-1-1.引用计数
引用计数:跟踪记录每个值被引用的次数，如果有一个值的引用次数为0，就表示这个值不再用到了，因此可以将这块内存释放。
原理:每次引用加一，被释放时减一，当这个值当引用次数变成0时，就可以将其内存空间回收。
示例:
const obj = {a:10}; //引用+1
const obj1 = {a:10}; //引用+1
const obj = {}; //引用-1
const obj = null; //引用为0
·声明了一个变量并将一个引用类型的值赋值给这个变量，这个引用类型的值的引用次数是1
·同一个值又被赋值给另外一个变量，这个引用类型值的引用次数加1
·当包含这个引用类型值的变量又被赋值成另外一个值了，难么这个引用类型值的引用次数减1
·当引用次数变成0时，说明没办法访问这个值了
·当垃圾收集器下一次运行时，它就会释放引用次数是0的值所占用的内存
引用计数的bug——循环引用(也就是相互引用),会造成js垃圾回收机制无法回收内存，浪费大量性能
function fn(){
var objA = {a:10};
var objB = {a:10};
objA.c = objA;
objB.d = objB;
}
在采用引用策略的实现中，当函数执行完毕后，objA和objB还将继续存在，因为它们的引用次数永远不会是0，假如这个函数被多次重复调用就会导致大量内存得不到回收。这个问题主要出现在IE8以及以前的时代，现在浏览器更多采用标记清除垃圾回收机制。
3-1-2.标记清除
(1)标记清除指的是当变量进入环境时，这个变量标记为“进入环境”；而当变量离开环境时，则将其标记为“离开环境”，最后，垃圾收集器完成内存清楚工作，销毁那些带标记的值并回收它们所占用的内存空间。
(2)上面说到的环境即为执行环境，执行环境:执行环境定义了变量或函数有权访问的其他数据，决定了它们各自的行为。每个执行环境都有一个与之关联的变量对象(variable object)，环境中定义的所有变量和函数都保存在这个对象中。
执行环境分为全局执行环境和局部执行环境，虽然我们在编程时无法访问这个变量，但解析器在处理数据时，会在后台使用它。
(2-1)全局执行环境
·最外围的一个执行环境
·根据宿主环境不同表示执行环境的对象也不一样。在浏览器中，全局的执行环境被认为是window对象(node环境是global)
·全局变量和函数都是作为window对象的属性和方法创建的
·某个执行环境中的所有代码执行完毕后，该环境被销毁，保存在其中的所有变量和函数定义也随之销毁(例如关闭网页或浏览器)
(2-2)环境栈(局部)
每个函数都有自己的执行环境。当执行流进入一个函数时，函数的环境就会被推入一个环境栈中。而在函数执行之后，栈将其环境弹出，把控制权返回给之前的执行环境。ECMAScript程序中的执行流正是由这个方便的机制控制着。
示例
function foo(){
var a = 10; //被标记进入环境
var b = ‘Hello’; //被标记进入环境
}
foo(); //执行完毕，a和b被标记离开环境，内存被回收
目前主流浏览器都使用标记清除垃圾回收策略
4.V8内存管理机制
4-1.V8限制内存的原因
·V8最初为浏览器而设计，不太可能遇到大量内存的使用场景
·防止因为垃圾回收所导致的线程暂停执行的时间过长
4-2.V8的回收策略
·V8采用了一种分代回收的策略，将内存分为两个生代:新生代和老生代
·V8分别对新生代和老生代使用不同的垃圾回收算法来提升垃圾回收的效率
(1-1)新生代垃圾回收

(1-2)新生代对象的晋升
·在新生代垃圾回收的过程中，当一个对象经过多次复制后依然存活，它将被认为是生命周期较长的对象，随后会被移动到老生代中，采用新的算法进行管理。
·在From空间和To空间进行反转的过程中，如果To空间中的使用量已经超过了25%，那么就将From中的对象直接晋升到老生代内存空间中
(1-3)老生代垃圾回收
老生代内存空间是一个连续的结构
(1-4)老生代垃圾清除
·标记清除(Mark Sweep):是将需要被回收的对象进行标记，在垃圾回收运行时直接释放相应的地址空间
·标记合并(Mark Compact):将存活的对象移动到一边，将需要被回收的对象移动到另一边，然后对需要被回收的对象区域进行整体的垃圾回收
