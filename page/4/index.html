2.受控组件/非受控组件
这两个都是针对于表单来说的，最明显的区别就是value和defaultValue的区别，如下图

还有就是受控组件，必须写value和onChange，这时候挡在输入框写内容的时候才会发生变化；非受控组件，只写defaultValue就可以了(受控组件如果不写onChange那么输入框呈现的值是不会发生变化的，非受控组件即使不写onChange输入框呈现的值会根据输入内容进行改变)。对于受控组件，数据和onChange函数是严格绑定在一起的，很严谨，出错的可能性比较小，但是需要我们将value和onChange同时写出来；非受控组件自由性更大，但是出错的可能性更高。综上，更建议用受控组件，非受控组件只作为一个了解项。

张容铭-设计模式-第四篇-行为设计模式

其实本来还没看到这里的，但是最近学习的东西有些与观察者模式相关，所以这里先记录观察者模式。
一、观察者模式
观察者模式:又被称作发布-订阅者模式或消息机制，定义了一种依赖关系，解决了主体对象于观察者之间功能的耦合。
假设我们现在有一颗卫星(观察者)，可以接收飞机飞行的信息，并且把这些信息发布到需要(订阅者)这个消息的地方。观察者对象有一个消息容器，和三个方法，分别是订阅消息方法，取消订阅的消息方法，发送订阅的消息方法。


//将观察者放在闭包中，当页面加载就立即执行
var Observer = (
    function(){
        //防止消息队列暴露而被篡改故意将消息容器作为静态私有变量保存
        var _messages = {};
        return {
            //注册信息接口
            regist : function(){},
            //发布信息接口
            fire : fuction(){},
            //移除信息接口
            remove : function(){}
        }
    }
)();

消息注册方法，该方法的作用是将订阅者注册的消息推入到消息队列中，因此我们需要接受两个参数:消息类型以及相应的处理动作，在推入到消息队列时如果此消息不存在则应创建一个该消息类型并将该消息放入消息队列中，如果此消息存在则应该将该消息执行方法推入该消息对应的执行方法队列中，这么做的目的也是保证多个模块注册同一则消息时能顺利执行。


regist : function(type,fn){
    //如果此消息不存在则应该创建一个该消息类型
    if(typeof _messages[type] === ‘undefined){
        //将动作推入到该消息对应的动作执行队列中
        _messages[type] = [fn];
    //如果此消息存在
    }else{
        //将动作方法推入该消息对应的动作执行序列中
        _messages[type].push(fn)
    }
}

对于发布消息方法，其功能是当观察者发布一个消息时将所有订阅者订阅的消息一次执行。故应接收两个参数，消息类型以及动作执行时需要传递的参数，当然在这里消息类型过是必须的。在执行消息动作队列之前校验消息的存在是很有必要的。然后遍历消息执行方法队列，并依次执行。然后将消息类别以及传递的参数打包后依次传入消息执行方法中。


fire : function(type,args){
    //如果该消息没有被注册，则返回
    if(!_messages[type]){
        return;
    }
    //定义消息信息
    var events = {
        type : type,                      //消息类型
        args : args || {}                 //消息携带数据
    },
    i = 0,                               //消息动作循环变量
    len = _messages[type].length;        //消息动作长度
    //遍历消息动作
    for( ; i < len ; i++){
        //依次执行注册的消息对应的动作序列
        _messages[type][i].call(this, events)
    }
}

最后是消息注销方法，其功能是将订阅者注销的消息从消息队列中清除，因此我们也需要两个参数，即消息类型以及执行的某一动作。当然为了避免删除消息动作时消息不存在的情况的出现，对消息队列中消息的存在性校验也是很有必要的


remove : function(type,fn){
    //如果消息动作队列存在
    if(_message[type] instanceof Array){
        //从最后一个消息动作遍历
        var i = _messages[type].length - 1;
        for( ; i >=0 ; i–){
            //如果存在该动作则在消息动作序列中移除相应动作
            _messages[type][i] === fn && _message[type].splice(i,1)
        }
    }
}

上面我们初步了解了观察者模式。但是观察者模式有什么作用？观察者模式有一个作用是解耦代码。下面我们更详细的了解一下
如果有一个这样的需求:一个新闻评论模块，当用户发布评论时，会在评论展示模块末尾处追加新的评论，与此同时用户的消息模块的消息数量也会递增。如果用户删除留言区的信息时，用户的消息模块消息数量也会递减，但是这些模块是三位不同的工程师写的，也就是说都写在自己独立闭包的模块里。如果要完成现在的需求，又不想将他们的模块合并到一起，该如何解决？
同一模块内的功能应该放在一起，这样管理起来菜会更加方便。所以对于追加留言的功能就应放在之前开发过的留言模块里；对于用户消息递增功能应当放在之前开发过的用户信息模块里。对于留言的提交，理应放在提交模块里。不过，既然我们选择观察者模式来解决问题，首先就要分析那些模块应该注册消息，那些模块应该发布消息。
发布留言与删除留言功能需求是用户主动触发，所以应该是观察者发布消息，而评论的追加以及用户消息的增减是被动触发的，所以他们应该是订阅者去注册消息。这样的话用户信息模块既是信息的发送者，也是信息的接收者，提交模块是信息的发送者，浏览模块是信息的接收者。

观察者模式最主要的作用是解决类或对象之间的耦合，解耦两个相互依赖的对象，使其依赖于观察者的消息机制。这样对于任意一个订阅者对象来说，其他订阅者对象的改变不会影响到自身。对于每一个订阅者来说，其自身既可以是消息的发出者也可以是消息的执行者，这都依赖于调用观察者对象的三种方法(订阅消息，注销消息，发布消息)中的哪一种。
团队开发中，尤其大型项目的模块开发中，每个人很难做到熟知项目中的每个模块，此时为完成一个涉及多个模块调用的需求，观察者模式的优势就显而易见了，模块间的信息传递不必要相互引用其他模块，只需要通过观察者模式注册或者发布消息即可。通过观察者模式，工程师间对功能的开发只需要按照给定的消息格式开发各自功能即可，而不必去担忧其他人的模块。

    on(name,cb) {

    }

        delete this.cb[name]
    }
    trigger(name,arg) {
        this.cb[name](arg)
    }
}

        })
    }

class Game extends React.Component {
  constructor(props){
    super(props);
    this.state = {
      history:[{
        squares:Array(9).fill(null)
      }],
      stepNumber:0,
      xIsNext:true,
    }
  }
  jumpTo(step) {
    this.setState({
      stepNumber: step,
      xIsNext: (step % 2) === 0,
    });
  }
  handleClick(i){
    const history = this.state.history.slice(0, this.state.stepNumber + 1);
    const current = history[history.length - 1];
    const squares = current.squares.slice();
    if (calculateWinner(squares) || squares[i]) {
      return;
    }
    squares[i] = this.state.xIsNext ? 'X' : 'O';
    // console.log(squares)
    this.setState({
      history: history.concat([{
        squares: squares,
      }]),
      stepNumber: history.length,
      xIsNext:!this.state.xIsNext,
    })
  }
  render() {
    const history = this.state.history;
    const current = history[this.state.stepNumber];
    const winner = calculateWinner(current.squares);

    const moves = history.map((step,move)=>{
      const desc = move ? 'Go to move #' + move : 'Go to game start';
      return (
        < li key = {move}>
        < button onClick={() => this.jumpTo(move)}>{desc}< /button>
        < /li>
      )
    })

    let status;
    if (winner) {
      status = 'Winner: ' + winner;
    } else {
      status = 'Next player: ' + (this.state.xIsNext ? 'X' : 'O');
    }
    return (
      < div className="game">
        < div className="game-board">
          < Board 
          squares = {current.squares}
          onClick = {(i)=>{this.handleClick(i)}}
          />
        < /div>
        < div className="game-info">
          < div>{status}< /div>
          < ol>{moves}< /ol>
        < /div>
      < /div>
    );
  }
}

function calculateWinner(squares){
  const lines = [
    [0, 1, 2],
    [3, 4, 5],
    [6, 7, 8],
    [0, 3, 6],
    [1, 4, 7],
    [2, 5, 8],
    [0, 4, 8],
    [2, 4, 6],
  ];
  for(let i=0;i < lines.length;i++){
    const [a,b,c] = lines[i];
    if (squares[a] && squares[a] === squares[b] && squares[a] === squares[c]) {
      return squares[a];
    }
  }
  return null;
}
